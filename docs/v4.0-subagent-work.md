## Phase 2: `modules/time_estimator.py` — 时长推算式

**目标：** 为不发 WS progress 事件的节点（如 SeedVR2VideoUpscaler）提供时长推算法。

**具体任务：**
1. 创建 `modules/time_estimator.py`
2. 实现 `TimeEstimator` 类：
   - `_history: dict[str, list[float]]` — 历史耗时记录
   - `record(node_class, elapsed)` — 出图完成后写入
   - `estimate(node_class, resolution_hint=0)` — 中位数估算
   - `progress(node_class, elapsed, resolution_hint=0)` — 返回 `(pct, expected)`
3. 分辨率加权默认值：4K→120s, 2K→60s, <2K→30s

**验证清单：**
- [ ] `TimeEstimator.estimate("SeedVR2VideoUpscaler", 4096)` 返回 120
- [ ] 调用 `record()` 后，`estimate()` 返回更新后的中位数
- [ ] `progress()` 在 elapsed > expected×1.5 时 clamp 到 95
- [ ] 线程安全（类变量字典操作）

## Phase 3: `modules/instance_manager.py` — 实例生命周期管理

**目标：** 集中管理实例冷启动、健康检查、空闲回收。

**具体任务：**
1. 创建 `modules/instance_manager.py`
2. 实现 `@dataclass InstanceHealth` 类
3. 实现 `InstanceManager` 类：
   - `__init__(nodes_provider)` — 接收获取实例列表的闭包
   - `async ensure_running(instance, timeout=300)` — 唯一冷启动入口
   - `async health(instance, force=False)` — 健康检查（缓存 15s）
   - `mark_active(name)` — 更新时间戳
   - `is_grace(name)` — 90s 防御期检查
   - `get_health_summary()` — 全量健康快照
   - `async start(instance)` / `stop(instance)` / `restart(instance)`
   - `async _dead_check_loop()` — 60s 间隔
   - `async _idle_reap_loop()` — 60s 间隔
4. 使用 `config.py` 中的常量

**验证清单：**
- [ ] `ensure_running()` 对已 running 实例立即返回
- [ ] `ensure_running()` 同实例并发调用只启动一次（去重锁）
- [ ] `health()` 缓存 15s，`force=True` 强制实时
- [ ] `is_grace()` 在启动后 90s 内返回 True
- [ ] `_dead_check_loop` 跳过 grace period 内的实例
- [ ] `_idle_reap_loop` 跳过有活跃 job 的实例

## Phase 4: `modules/step_calculator.py` — 进度计算引擎

**目标：** 从 workflow JSON 计算 total_units 和各节点权重。

**具体任务：**
1. 创建 `modules/step_calculator.py`
2. 实现 `@dataclass StepInfo`：
   - `total_units: int`
   - `node_weights: dict[str, float]` — node_id→权重（0=时长推算）
   - `node_labels: dict[str, str]` — node_id→可读标签
   - `sampler_steps: dict[str, int]` — node_id→有效步数
   - `node_order: list[str]` — 拓扑排序
   - `time_estimates: dict[str, float]` — node_id→预计耗时
3. 实现 `StepCalculator` 类：
   - `calculate(workflow)` — 主入口
   - `resolve_steps(node_id, workflow, depth=0)` — 递归链路解析
   - `_resolve_steps_or_default()` / `_resolve_float_or_default()`
   - `get_category(class_type)` — 返回分类名
4. SAMPLER weight = `steps × denoise + 1`，含递归链路解析
5. VAEDecode weight = 1（修复缺陷 A）
6. 无 steps 节点 weight = 0 → 写入 `time_estimates`

**验证清单：**
- [ ] `calculate()` 对 t2i-z-image.json 返回正确的 total_units
- [ ] `resolve_steps()` 能解析 `['157', 0]` 类型的链接
- [ ] VAEDecode 权重 = 1（不再跳过）
- [ ] 有 denoise 的采样器 weight = `steps × denoise + 1`
- [ ] SeedVR2VideoUpscaler 的 weight = 0, 有 time_estimates
- [ ] `get_category("KSampler")` → "sampler"

## Code Review Checklist（每阶段必查）

### 通用
- [ ] 无 `print()` / `traceback.print_exc()` 语句（除非日志替代）
- [ ] 所有异常有适当的 try/except，不会静默失败
- [ ] 不引入新的第三方依赖
- [ ] 类型 hint 完整（参数 + 返回值）
- [ ] 文档字符串（docstring）清晰描述输入/输出/副作用

### 模块化
- [ ] 无循环 import（A→B→A）
- [ ] 模块之间按 §4 依赖图单向引用
- [ ] `modules/__init__.py` 只做 re-export，不做业务逻辑
- [ ] 所有 IO/网络/进程调用集中在 instance_manager 和 ws_tracker

### 正确性
- [ ] `app.py` 启动后现有 API 路由不变
- [ ] `app.py` 启动后前端页面不报错
- [ ] WS connection/disconnect 不做资源泄漏
- [ ] asyncio 协程正确创建和取消（无 dangling task）
- [ ] 进度 pct 永不回退（单调递增保证）

### 健壮性
- [ ] `ensure_running()` 有超时机制（默认 300s）
- [ ] WS connect 有重试机制（3 次）
- [ ] HTTP fallback 有重试机制
- [ ] `cancel()` 正确释放所有信号量
- [ ] 所有 `asyncio.Semaphore` acquire 有 timeout
