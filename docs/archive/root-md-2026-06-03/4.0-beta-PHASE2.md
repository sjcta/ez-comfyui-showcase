# 4.0-beta Phase 2 — 架构重构

## 概述
三个独立子任务，可并行执行。

---

## Task A: 局部更新修复全屏闪烁

### 问题
实例启停操作后调用 `loadNodes()` 全量刷新渲染整个设备列表，导致闪烁。

### 方案
启停操作后，只更新受影响的那一条实例行，不刷新整个列表。

### 改动文件
- `static/js/modules/nodes.js`

### 具体改动

**现状**（以 stopInstance 为例）：
```javascript
async function stopInstance(nid, iid) {
  ...
  loadNodes();  // ← 全量刷新，导致闪烁
}
```

**改为**：
1. `startInstance` / `stopInstance` / `restartInstance` 成功后不调用 `loadNodes()`
2. 而是调用 `updateInstanceRow(nid, iid, newStatus)`：
   - 找到当前卡片中该实例的行
   - 只更新状态灯颜色、状态文字、按钮组（从"停止"切换到"启动"，反之亦然）
   - 队列数字如果有变化也更新
3. 新增 `updateInstanceRow(nid, iid)` 函数内部逻辑：
   - 从 API 获取最新状态：`GET /api/nodes/{nid}`
   - 找到该实例的最新状态
   - 通过 DOM 操作更新该行
4. 其他按钮（编辑、删除、测试连通、扫描端口）保持调用 `loadNodes()`——这些操作不频繁且需要数据刷新

### 约束
- 不要破坏现有功能
- 仅改启停操作的刷新逻辑，其他不变

## Task B: 工作流仓库架构

### 问题
1. 同步/上传/目录管理混合在同一个面板区，设计混乱
2. 目录管理已无实际作用
3. 工作流存放路径无结构

### 新架构

**目录结构**：
```
{base_dir}/                          ← 用户配置的工作流基础目录（默认 data/workflows/）
├── {device_name}/                   ← 远程同步（自动创建，来源设备名）
│   ├── t2i-z-image.json
│   └── i2i-Qwen-Edit.json
├── ez-comfy/                        ← 手动上传（固定目录）
│   ├── my-custom-wf.json
│   └── another.json
└── ...                              ← 用户手动放置的保持原样
```

**元数据追踪**：
`data/wf_meta.json` 扩展字段：
```json
{
  "t2i-z-image.json": {
    "name": "Z-Image-Turbo 标准",
    "tags": ["文生图", "2K"],
    "source": "DGX Spark",        // 来源设备名，ez-comfy 上传的为 "local"
    "source_path": "/base_dir/DGX Spark/t2i-z-image.json",
    "synced_at": "2026-05-12T16:00:00Z"
  }
}
```

### 工作流管理 UI 改动

**分离为三个区域**：

| 区域 | 内容 |
|------|------|
| 📂 我的工作流 | 工作流卡片网格（和现在一样，不改变） |
| 📥 远程同步 | 设备选择器 + 同步按钮 + 同步状态/日志 |
| 📤 手动上传 | 拖入/选择文件上传（和现在上传区类似） |

**移除**：旧的「工作流目录」管理部分（`wfDirsSection`）

### 改动文件
- `index.html` — 重构工作流管理面板底部区域
- `style.css` — 新增区域样式
- `workflows.js` — 同步/上传逻辑分离
- `app.py` — `sync_remote_workflows()` 写入路径改为 `{base_dir}/{device_name}/`
- `app.py` — 上传 `POST /api/workflows/upload` 写入路径改为 `{base_dir}/ez-comfy/`

## Task C: 出图历史 SQLite 数据库

### 问题
当前出图历史存储在 `jobs.json` 中，查询/持久化能力弱。

### 方案
建立 SQLite 数据库 `data/generation.db`

### 表结构

```sql
CREATE TABLE IF NOT EXISTS generations (
  id TEXT PRIMARY KEY,
  workflow TEXT NOT NULL,
  workflow_name TEXT DEFAULT '',
  device TEXT DEFAULT '',
  instance TEXT DEFAULT '',
  status TEXT DEFAULT 'done',
  image_path TEXT DEFAULT '',
  thumb_path TEXT DEFAULT '',
  created_at DATETIME DEFAULT (datetime('now','localtime')),
  completed_at DATETIME,
  duration_sec INTEGER DEFAULT 0,
  params TEXT DEFAULT '{}',
  prompt TEXT DEFAULT '',
  width INTEGER DEFAULT 0,
  height INTEGER DEFAULT 0,
  seed INTEGER DEFAULT 0
);
```

### API 端点

| 方法 | 路径 | 用途 |
|------|------|------|
| GET | /api/history | 分页查询（limit/offset/status 参数） |
| POST | /api/history | 插入记录 |
| DELETE | /api/history/{id} | 删除记录 |
| DELETE | /api/history | 清空全部 |

### 迁移
- 保留现有 `jobs.json` 读取逻辑作为 fallback
- 新生成的记录写入 SQLite
- 后续可做一步迁移脚本将 jobs.json 导入 SQLite

### 前端
- `history.js` 中的 `loadHistory()` 改为调用 `GET /api/history`
- 兼容现有卡片渲染逻辑不变

### 改动文件
- `app.py` — 新增数据库模块 + API 端点
- `data/generation.db` — 自动创建（首次启动时）
- `history.js` — 修改数据源为 API
