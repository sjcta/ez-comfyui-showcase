# Ez ComfyUI v4.0 — 出图 Pipeline 重构设计

> 编写日期: 2026-05-15
> 状态: 设计阶段
> 目标版本: v4.0 (重构后)

---

## 1. 设计目标

### 1.1 消灭三重冷启动
当前一个出图任务最多可能等待 **660 秒** 的冷启动时间却没有实质性进展。重构后冷启动只发生 **一次**。

### 1.2 实例并行化
移除全局信号量，以实例级别信号量控制并发，做到 A/B 实例真正并行出图。

### 1.3 前端无闪烁
将卡片状态切换从 "全量 rebuild" 改为 "数据驱动 + 局部 patch"，消除每次状态变化的闪屏。

### 1.4 统一决策链
实例健康检查、冷启动触发、空闲回收收敛到单一管理器。

### 3.3 模块契约：`modules/step_calculator.py`

**职责：** 从 workflow JSON 解析节点拓扑，正确计算 `total_units` 和各节点权重。

```python
@dataclass
class StepInfo:
    """一次性计算结果，供 ws_tracker 全程引用"""
    total_units: int
    node_weights: dict[str, float]   # node_id → unit_count (0=仅时长推算)
    node_labels: dict[str, str]      # node_id → "采样中..."
    sampler_steps: dict[str, int]    # node_id → effective_steps (采样器/超分器)
    node_order: list[str]            # 拓扑排序
    time_estimates: dict[str, float] # node_id → expected_seconds (仅 weight=0 的节点)

class StepCalculator:
    """
    输入: workflow dict (ComfyUI prompt 格式)
    输出: StepInfo dataclass
    使用: config.NodeCategory 做节点分类
    """

    def calculate(self, workflow: dict) -> StepInfo:
        """
        遍历所有节点，逐类分配权重：
        - SAMPLER:  steps × denoise + 1  (初始化)
        - UPSCALE:  steps + 1 (若 steps=0 走 TimeEstimator)
        - FREE:     weight=0, 不占进度
        - LOADER:   weight=1
        - WEIGHT_1: weight=1
        - 未归类:   weight=1
        - 无 steps 参数: weight=0, 写入 time_estimates
        """

    def resolve_steps(self, node_id: str, workflow: dict, depth: int = 0) -> int:
        """递归解析 steps 参数，至多 5 层，follow ComfySwitchNode/PrimitiveInt"""

    @staticmethod
    def get_category(class_type: str) -> str:
        """返回 'sampler' | 'upscale' | 'free' | 'normal' | 'time_estimate'"""
```

**删除的旧代码：** `comfyui_ws_track()` 中 L1417-L1443 的 total_units 计算逻辑（移入此模块）

**依赖：** `config.py`, `time_estimator.py`

**扩展点：** 新增 workflow 类型 = 先确认节点分类是否在 `config.py` 中覆盖，若没有则添加。

### 3.4 模块契约：`modules/time_estimator.py`

**职责：** 为不发 WS progress 事件的节点（如 SeedVR2VideoUpscaler）做时长推算。

```python
class TimeEstimator:
    """
    输入: node_class + resolution_hint + elapsed
    输出: (pct_in_node: 0~95, expected_seconds)
    数据: 历史已完成任务的真实耗时（全局共享）
    """

    # 类变量：history = { node_class: [elapsed_seconds] }
    _history: dict[str, list[float]] = {}
    _max_samples = 20

    @classmethod
    def record(cls, node_class: str, elapsed: float):
        """出图完成后由 job_runner 回调写入"""

    @classmethod
    def estimate(cls, node_class: str, resolution_hint: int = 0) -> float:
        """
        中位数法估算（抗 outliers）
        无历史时按分辨率硬编码默认值：
          4K upscale → 120s, 2K → 60s, <2K → 30s
        """

    @classmethod
    def progress(cls, node_class: str, elapsed: float,
                 resolution_hint: int = 0) -> tuple[float, float]:
        """
        pct = min(elapsed / expected * 100, 95)
        clamp 在 95%, 留 5% 给 completing 状态
        """
```

**依赖：** 无

**扩展点：** 如需支持新的无声节点，只需 `TimeEstimator.record()` 会自然积累数据，无需配置。

### 3.5 模块契约：`modules/ws_tracker.py`

**职责：** WebSocket 通信 + 实时进度追踪 + HTTP fallback + 断线重连。

```python
@dataclass
class TrackResult:
    ok: bool           # 出图是否成功
    prompt_id: str     # ComfyUI prompt_id
    elapsed: float     # 总耗时（秒）

class WSTracker:
    """
    输入: job_id, workflow dict, StepInfo, instance url
    输出: TrackResult
    """

    def __init__(self, job_id: str, workflow: dict, step_info: StepInfo,
                 instance_url: str, node_types: dict[str, str]):
        self._progress_callback: Callable[[dict], Awaitable[None]] | None = None
        # 外部注入 broadcast 函数

    async def track(self, timeout: int = 900) -> TrackResult:
        """
        流程:
        1. WS connect (重试 3 次)
        2. POST /prompt 提交 workflow
        3. while not timeout:
             - WS 收到 executing/progress: 更新 completed_units, 调 callback
             - WS 断连: 退化 HTTP poll /history/{pid}
             - WS 无消息 300s: 退化 HTTP
             - WS 收到 execution_error: raise RuntimeError
             - WS 收到 executing:null: 完成
        4. 返回 TrackResult
        """

    def get_current_progress(self) -> dict:
        """返回 { pct: float, message: str, current_node: str } 供广播"""
```

**扩展点：** WS 协议升级（如 ComfyUI 未来版本改消息格式）只需要改 `_parse_message()` 内部逻辑。

### 3.9 前端模块契约：`static/js/modules/poll_manager.js`

**职责：** 统一管理 WS + HTTP fallback 两个数据源，避免冲突。

```javascript
/**
 * PollManager — WS 优先, HTTP polling 兜底。
 *
 * 数据流:
 *   WS onmessage → onJobUpdate(job)   ← 实时
 *   HTTP GET /api/jobs (3s) → merge  ← fallback
 *
 * 合并规则:
 *   - 同 id → 后者覆盖（仅当 status 改变或 progress 刷新）
 *   - WS 数据比 HTTP 更新 → WS 的 onJobUpdate 优先
 *   - HTTP 仅做"没收到 WS 推送时的保底"
 */
export class PollManager {
    constructor(jobs, broadcastFn);

    /** 启动 WS + HTTP polling */
    start() → void;

    /** 停止 polling（页面关闭时） */
    stop() → void;

    /** WS 推送回调 */
    onJobUpdate(job) → void;
    // 1. 更新 jobs[id]
    // 2. status 变 → CardManager.renderGallery()
    // 3. progress 变 → CardManager.patchJobCard()

    /** HTTP fallback 回调 */
    _httpPoll() → void;
    // 1. GET /api/jobs
    // 2. 与 jobs 做 diff
    // 3. 有差异 → 同 onJobUpdate 路径
}
```

**删除的旧代码：** `pollJobs()` (5s) 全量轮询；客户端 700s 安全逻辑（信任服务端 stuck watcher）

**依赖：** `card_manager.js`

```python
if cls != "VAEDecode":
    non_sampler_cnt += 1

# 运行时计数（当前 L1506）:
if cls not in SAMPLER_NODES and cls not in UPSCALE_ACT_NODES and cls != "VAEDecode":
    completed_units += 1.0
```

VAEDecode 在两端都被排除，consistent but invisible。用户看到采样 `20/20` 进度的瞬间，进度条卡在 ~95%，然后 VAE 解码时不动，最后突然跳到 100%。

**修复：** VAEDecode 计入 `non_sampler_cnt` 和 `completed_units`（权重=1 unit）。

---

**缺陷 B：Sampler 初始化占用 1 unit 但未计入 total**

```python
# executing 事件处理（当前 L1510）:
completed_units += 1.0  # sampler 开始执行时加 1 unit
```

total_units 只加了 `steps` 的数量（例如 20），但 sampler 启动时额外消耗了 1 个 unit；
结果进度超过 100%（21/20），前端被迫 clamp 在 100%。

**修复：** 每个 sampler/upscaler 的 `sampler_steps_total` 加上 1（初始化 unit）：

```python
sampler_steps_total += int(v) + 1  # +1 给初始化事件
```

---

**缺陷 C：`denoise` 影响实际步数但未考虑**

```python
# 当前 L1425:
v = inp.get("steps", 8 if cls in SAMPLER_NODES else 4)
sampler_steps_total += int(v)
```

若 `steps=20` 且 `denoise=0.5`，ComfyUI 实际执行 10-12 步。total_units 按 20+1 算，
但 progress 事件只发 10 次，进度条只走到一半就跳到下一阶段。

**修复：** 计算 effective steps 时乘以 denoise：

```python
denoise = inp.get("denoise", 1.0)
if isinstance(denoise, list): denoise = 1.0  # 链接节点，无法提前解析
if isinstance(raw_steps, (int, float)):
    effective_steps = max(1, int(raw_steps * denoise))
sampler_steps_total += effective_steps + 1  # +1 给初始化
```

---

**缺陷 D：`steps` 参数链接到其他节点时解析不完整**

```python
# 当前 L1427:
elif isinstance(v, list) and len(v) >= 1:
    ln = workflow.get(str(v[0]), {}).get("inputs", {})
    for k in ("value", "INT"):
        if k in ln and isinstance(ln[k], (int, float)):
            sampler_steps_total += int(ln[k])
```

只检查直接链路的 `value`/`INT`。遇 ComfySwitchNode、多层链路时走 fallback（8），不准确。

**修复：** 使用递归链路解析器 `resolve_steps(node_id, workflow, depth=0)`，
至多跟 5 层，找到具体数值：

- PrimitiveInt → `inputs.value`
- ComfySwitchNode → 检查 on_true/on_false 分支
- 其他 → 逐层 follow
- 超 5 层 → fallback 8

---

**缺陷 E：N 图并行出图（batch_size>1 或 n>1）**

某些 workflow 每次生成 N 张图（重复 KSampler 或 latent_image 的 batch_size/batch_count）。
当前代码按"单图"计算 total_units，但实际收到 N 倍 progress 事件。
进度条跑完一次 100% 后还在继续接收事件，造成 100% → 停滞 → 又动几下。

**修复**（二选一）：
a) WS 事件处理中若 `prompt_id` 已在 history 中出现，忽略剩余 progress 事件。
b) 在 StepCalculator 中检测重复采样器结构，按 `重复次数 × steps` 调整 total_units。

### 5.2 StepCalculator 模块设计（已落地 `modules/step_calculator.py`）

参考 §3.3 中 `StepCalculator` 接口。

```python
class StepCalculator:
    SAMPLER_CLASSES = {"KSampler", "KSamplerAdvanced", "SamplerCustom", "FluxSampler"}
    UPSCALE_CLASSES = {"ImageUpscaleWithModel", "SeedVR2VideoUpscaler"}
    FREE_CLASSES = {"SaveImage", "PreviewImage"}
    WEIGHT_1 = { /* 见 config.py */ }

    def resolve_steps(self, node_id, workflow, depth=0) -> int:
        """递归解析 steps，至多 5 层"""
        if depth > 5: return 8
        node = workflow.get(node_id)
        if not isinstance(node, dict): return 8
        inputs = node.get("inputs", {})
        ct = node.get("class_type", "")
        if ct == "PrimitiveInt":
            return int(inputs.get("value", 8))
        if ct == "ComfySwitchNode":
            for branch in ("on_true", "on_false"):
                if branch in inputs:
                    val = self.resolve_steps(str(inputs[branch][0]), workflow, depth + 1)
                    if val: return val
        return 8

    def calculate(self, workflow: dict) -> StepInfo:
        """遍历节点，逐类分配权重"""
        result = StepInfo(total_units=0, node_weights={}, ...)
        for nid, node in sorted(workflow.items(), ...):
            ct = node.get("class_type", "")
            if ct in self.FREE_CLASSES: continue
            if ct in self.SAMPLER_CLASSES:
                raw_steps = self._resolve_steps_or_default(node, "steps", 8, workflow)
                denoise = self._resolve_float_or_default(node, "denoise", 1.0)
                effective = max(1, int(raw_steps * denoise))
                result.total_units += effective + 1
                result.node_weights[nid] = float(effective + 1)
            elif ct in self.UPSCALE_CLASSES:
                raw_steps = self._resolve_steps_or_default(node, "steps", 4, workflow)
                if raw_steps > 0:
                    result.total_units += raw_steps + 1
                    result.node_weights[nid] = float(raw_steps + 1)
                else:
                    # weight=0 → time_estimate
                    result.node_weights[nid] = 0.0
                    result.time_estimates[nid] = TimeEstimator.estimate(ct, resolution)
            else:
                result.total_units += 1
                result.node_weights[nid] = 1.0
        return result

    def _resolve_steps_or_default(self, node, key, default, workflow):
        v = node.get("inputs", {}).get(key, default)
        if isinstance(v, list):
            return self.resolve_steps(str(v[0]), workflow)
        if isinstance(v, (int, float)):
            return int(v)
        return default

    def _resolve_float_or_default(self, node, key, default):
        v = node.get("inputs", {}).get(key, default)
        if isinstance(v, list): return default  # 链接节点不可提前解析
        return float(v)
```

### 5.3 WS 运行时进度同步

```python
async def comfyui_ws_track(job_id, workflow, client_id, step_info, ...):
    total_units = step_info.total_units
    node_weights = step_info.node_weights
    completed_units = 0.0
    last_prog = 0
    node_entered_at = {}  # 仅 weight=0 的节点

    while not timeout:
        raw = await ws.recv()
        msg = json.loads(raw)

        if msg["type"] == "executing":
            node_id = msg["data"].get("node")
            if node_id is None:  # 完成
                completed_units = total_units
                break
            nid = str(node_id)
            weight = node_weights.get(nid, 1.0)
            if weight > 0:
                completed_units += weight
            else:
                node_entered_at[nid] = time.time()  # 时长推算

        elif msg["type"] == "progress":
            cur, total = msg["data"]["value"], msg["data"]["max"]
            if cur > last_prog:
                node_weight = node_weights.get(str(msg["data"].get("node", "")), 1.0)
                if node_weight > 0:
                    completed_units += (cur - last_prog) / total * node_weight
                else:
                    # weight=0 节点也有 progress 事件
                    completed_units += (cur - last_prog) / total * 1.0
                last_prog = cur

        # 每 5s 刷新时长推算节点
        if int(time.time()) % 5 == 0:
            for nid, entered in node_entered_at.items():
                cls = node_types.get(nid, "")
                elapsed = time.time() - entered
                pct, _ = TimeEstimator.progress(cls, elapsed, resolution_hint)
                completed_units = max(completed_units, entered_base + pct / 100)

    pct = min(100, completed_units / total_units * 100) if total_units > 0 else 0
```

### 5.4 按 Workflow 类型的进度策略

##### 分类 A：标准 T2I（单采样器 + 单 VAE）

代表：`t2i-z-image.json`, `t2i-nunchaku-z-image-turbo-highSD.json`

```
模型加载 → CLIPTextEncode → EmptyLatentImage → KSampler(steps×denoise) → VAEDecode → SaveImage
```

- 模型加载类: 各 1 unit
- CLIPTextEncode: 1 unit；EmptyLatentImage: 1 unit
- KSampler: steps × denoise + 1 unit
- VAEDecode: 1 unit（修复缺陷 A，不再跳过）
- **示例**（8 steps, denoise=1）：加载模型 ~8% → CLIP ~16% → 潜空间 ~24% → 采样 1/8 ~37% → 8/8 ~90% → VAE 解码 ~98% → SaveImage 100%

##### 分类 B：高清放大 Upscale

代表：`SeedVR2_upscale_2k.json`, `SeedVR2_upscale_4k.json`

- SeedVR2VideoUpscaler 无 steps 参数，WS 无 progress 事件
- **策略**：退化为 `TimeEstimator` 时长推算法
- 进度上限 clamp 85%，完成时跳 100%

##### 分类 C：I2I 编辑

代表：`i2i-FireRed-Edit-8step.json`, `i2i-Qwen-Edit-v2511.json`

- 同 T2I，多出 LoadImage + VAEEncode + 额外 TextEncode
- KSampler 的 steps 通过链路解析读取

##### 分类 D：先 T2I 后 Upscale 级联

代表：`t2i-z-image-seedvr4k.json`, `t2i_nunchaku_seedvr4k.json`

- T2I 部分按 A 类，Upscale 部分按 B 类（时长推算）
- 中间不跳步，StepCalculator 全部计入

##### 分类 E：快速风格迁移（高降噪率）

代表：`t2i-z-image.json` 第二采样器（steps=2, denoise=0.25）

- weight = 2×0.25+1 = 1.5 unit
- WS progress 按完整 2 步发送，归一化后自动正确

| 类型 | total_units 计算 | progress 事件 | 退路 |
|------|-----------------|--------------|------|
| 标准 T2I | steps×denoise + 节点数 | ✅ 有 | — |
| Upscale | 时长估算 | ❌ 无 | 85%上限等待 |
| I2I | steps×denoise + 节点数 | ✅ 有 | — |
| 级联 | 混合 | 前半有后半无 | 后半退时长 |
| 风格迁移 | steps×denoise (低值) | ✅ 有 | — |

### 5.5 无 Steps 节点的进度推算（TimeEstimator）

参考 §3.4 中 `TimeEstimator` 接口。核心逻辑：

```python
@classmethod
def progress(cls, node_class, elapsed, resolution_hint=0):
    expected = cls.estimate(node_class, resolution_hint)
    if expected <= 0: return (0, 0)
    return (min(elapsed / expected * 100, 95), expected)
```

WS track 集成：weight=0 的节点进入时不加 unit，由 5s 定时器按 `TimeEstimator.progress()` 累加。

### 5.6 卡机、断流、超时的应急策略

| 场景 | 现象 | 策略 |
|------|------|------|
| WS 第一次连接失败 | ConnectionRefusedError | 3 次重试，间隔 2s；全退 HTTP polling |
| WS 中途断连 | ConnectionClosed | 记录已收进度，退化 HTTP poll /history/{pid} |
| WS 无消息超时 (300s) | asyncio.TimeoutError | 退化 HTTP |
| HTTP polling 也失败 | history 空 / network error | 保持 generating，不切 error；stuck watcher 兜底 |
| ComfyUI 存活但进度停滞 | WS 无消息，/system_stats 正常 | 标记 prolonged_stall，继续 poll 15min |
| ComfyUI 进程挂死 | /system_stats 挂 3 次 | 标记 error + 自动 restart 实例 |
| HTTP polling 返回 completed | WS 断了但正常跑完 | 正常出图，不惊动用户 |

**退化链路优先级：**

```
WS connected ────────────→ WS 实时进度
    │
    ├── WS 断连 ──→ HTTP poll /history/{pid} (3s)
    │                   ├── completed → 正常出图
    │                   ├── error → 标记 error
    │                   └── running → 继续等
    │
    ├── WS 连接失败 ──→ HTTP prompt (POST) + poll (3s)
    │
    └── HTTP 全失败 ──→ 保持 generating，等 stuck watcher (10min)
```

**原则：** 绝不因 WS 断连放弃任务。ComfyUI 服务端渲染继续，只是客户端收不到实时进度。

### 5.7 中间取消、重试、异常退出策略

**取消（Cancel）：**
1. status=generating → POST /interrupt
2. cancel WS track 协程
3. 释放 instance semaphore
4. 删除 jobs 条目，broadcast job_cancelled
5. interrupt 后需继续 poll 直到 prompt 在 history 中出现

**重试（Retry）：**
- error 卡片保留 60s + 重试按钮
- 点击 → 新 job_id + 新 seed + 继承参数 → 重新排队

**异常退出（实例死）：**
- 出图中的 job 标记 error + "实例意外停止，请重试"
- 自动 restart 实例

**Stuck Watcher 兜底（10 分钟）：**
- generating 状态 > 10min 无变化 → 标记 error + restart 实例

## 9. 边界条件与防御

### 9.1 冷启动去重
`ensure_running()` 通过 `asyncio.Lock` 确保同实例只启动一次，并发调用者等待。

### 9.2 Grace Period
启动后 90 秒内 `_dead_check_loop` 不误判。90 秒后仍未 up 则 restart。

### 9.3 空闲不打断
`_idle_reap_loop` 在 stop 前检查 instance sem 是否锁定，锁定中跳过。

### 9.4 Poll 不与 WS 冲突
WS 推送优先，3s polling 仅做 fallback。两者通过同一渲染队列安全合并。

### 9.5 Error 卡片保留
出图失败不删卡片。保留 error 状态 + 重试按钮，60s 后自动清理。
